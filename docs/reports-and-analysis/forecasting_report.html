1. System Overview

The AxBi Forecasting Engine is a multi-model time-series competition system accessible from the AI Insights page. The user selects a processed dataset, picks a time column, a numeric target, and optional feature columns. The backend loads the cleaned Parquet artifact, runs a 7-model competition with 3-fold rolling cross-validation, selects the winner by average rank, and returns the forecast with confidence intervals and full diagnostics.

2. The 7 Models

All models compete on every request. The winner is chosen by average rank across all four metrics — preventing a model that excels on one metric but fails on others from winning unfairly. If a runner-up scores within 25% of the winner on MAE, a 50/50 ensemble is used.

Fit Diagnosis — Overfitting Detection

Every model gets a fit diagnosis based on the ratio Test MAE ÷ CV MAE:

Label Ratio Range Meaning
Easy test split < 0.70× Test period was unusually easy — accuracy may be inflated
Healthy 0.70 – 1.20× Model generalizes well to unseen data
Mild overfit 1.20 – 1.50× Slight memorization — minor accuracy loss on holdout
Overfit 1.50 – 2.00× Model memorized training spikes, struggles on holdout
Severe overfit > 2.00× Unreliable — deprioritized by ensemble selection

Accuracy Metric

The displayed Accuracy % is computed as 100% − WAPE, where WAPE = Weighted Absolute Percentage Error = total absolute error ÷ total actual values. This gives a fair measure even when actuals are near zero. Color thresholds: green ≥ 95%, amber ≥ 85%, red < 85%.

5. Failure Cases & Fixes Applied

BUG — FIXED Prophet: "ufunc 'isnan' not supported for the input types"
Severity Critical — Prophet failed on ALL datasets
Root Cause pandas 2.x resample().agg() returns nullable Float64 (ExtensionArray). Prophet's internals call np.isnan(df['y'].values) which fails because numpy's isnan ufunc does not support pandas nullable Extension types. Assigning numpy arrays back to a DataFrame column gets re-wrapped as nullable type — requiring the fix at the DataFrame construction level.
Fix (1) In _prepare_series_frame: after resample().agg(), immediately cast all numeric columns via .to_numpy(dtype=np.float64, na_value=np.nan) before any further processing.
Fix (2) In ProphetForecaster.fit(): build the training DataFrame from a plain Python dict of numpy arrays — pd.to_numeric(series).to_numpy(dtype=np.float64, na_value=np.nan) — so pandas never infers nullable types during DataFrame construction.
Status ✓ Confirmed fixed — Prophet runs successfully on all 4 datasets
BUG — FIXED LightGBM: "pandas dtypes must be int, float or bool — bad dtype: string"
Severity High — LightGBM failed on datasets with categorical feature columns
Root Cause LightGBM requires all-numeric input. When the user selects a categorical column (e.g. product_category, channel) as a feature, it is passed as pandas string dtype and LightGBM rejects it at fit time.
Fix Added _encode_for_lgb() method using pd.factorize() to label-encode string columns before fitting. The encoding map is stored in self._lgb_label_maps and applied consistently during recursive prediction via _predict_recursive.
Status ✓ Confirmed fixed — LightGBM runs on all 4 datasets including those with categoricals
BUG — FIXED Step 7 Pipeline: "Gemini returned invalid JSON — Expecting ',' delimiter"
Affected HR dataset — step7_dashboard_blueprint.py
Root Cause The HR dataset has many columns with rich metadata. Full technical_stats + ai_profile payloads inflated the prompt, pushing the Gemini response beyond max_output_tokens: 4096. The response was truncated mid-JSON at char 4346, causing parse failure.
Fix (1) Increased max_output_tokens from 4096 → 8192.
Fix (2) Added response_mime_type: "application/json" on the first call (not just retry), forcing structured JSON output from Gemini.
Fix (3) Trimmed column context: send only name, data_type, is_primary_metric, and slim stats (min/max/mean/top_5) instead of full metadata blobs.
Status ✓ HR pipeline completes successfully
ISSUE — FIXED Gemini Model Chain: "gemini-3-flash-preview" does not exist
Affected Every Step 7 run — GEMINI_MODEL_CHAIN in step7_dashboard_blueprint.py
Root Cause gemini-3-flash-preview is not a valid Gemini model ID. Every Step 7 run silently errored on the 3rd fallback attempt and logged a warning unnecessarily.
Fix Replaced with gemini-1.5-flash as a stable, existing final fallback model.
Status ✓ Fixed — clean model chain: 2.5-flash → 2.5-pro → 1.5-flash
