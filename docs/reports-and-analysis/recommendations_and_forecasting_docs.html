Part 1

Recommendation Engine

Converts raw analytics outputs — AI report, forecasts, and segmentation results — into 3–5 prioritized, actionable business recommendations. Runs on-demand with intelligent caching.

Architecture Overview

The engine is a 5-stage pipeline implemented in backend/api/recommendations/. Each stage is a separate module with a single responsibility.

↓

↓

Stage 2 — Snapshot Hash & Cache

File: recommendations/service.py → _compute_snapshot_hash(ctx)

parts = [
    forecast.id  or  forecast.created_at,
    segmentation.generated_at  or  segmentation.method,
    step8.generated_at,
    step7.generated_at,
]
hash = sha1("|".join(parts))[:16 chars]

If global_context.recommendations.snapshot_hash matches the computed hash, the cached blob is returned with no Gemini API call. The cache is automatically invalidated when any of the following happen:

  • A new forecast run completes (new ID or timestamp)
  • Segmentation is re-run (new generated_at)
  • The pipeline re-generates the Step 8 report

Passing force=True bypasses the cache entirely and always regenerates.

Stage 3 — Signal Detectors

File: recommendations/signal_detectors.py

14 deterministic Python functions. Each inspects the context dict and returns a typed Signal dict or None. All run in sequence via run_all_detectors(ctx) — every detector that fires produces a signal. There is no early exit; all signals are collected.

Signal Schema

{
    "type":            str,    # identifier used in the prompt and frontend labels
    "severity":        str,    # "critical" | "warning" | "info"
    "affected_entity": str,    # column name, segment name, or filename
    "confidence":      float,  # 0.0 – 1.0 (used by Gemini for priority)
    "evidence": {
        # raw numbers that Gemini must cite in the recommendation text
        "metric":     "...",
        "value_pct":  40.5,
        ...
    }
}
Execution order matters: detect_step8_insights is listed first in the DETECTORS registry so Gemini always receives the AI report content as a base signal, even when no forecast or segmentation exists.

Stage 4 — Gemini Prompt & Call

Files: recommendations/prompts.py and recommendations/gemini_client.py

If at least one signal fired, build_prompt(ctx, signals) constructs the prompt and call_gemini(prompt) sends it.

Prompt Variables

  • category — domain language mapping (Sales → "customer", HR → "employee", Operations → "product", Marketing → "campaign")
  • filename + row_count — dataset identity for context
  • signals_json — full JSON array of all fired signals including their evidence dicts

What Gemini Returns

{
  "recommendations": [
    {
      "title":        "Imperative action phrase (max 8 words)",
      "rationale":    "1-2 sentences citing numbers from signal evidence",
      "priority":     "high | medium | low",
      "actions":      ["Concrete step 1", "Concrete step 2", ...],
      "metrics":      { "metric_name": 40.5 },   // real numbers only
      "triggered_by": ["signal_type", ...]
    }
  ]
}

Hard Prompt Rules

  • metrics values must be actual numbers from the signal evidence — never strings like "Track weekly" or "N/A"
  • Priority maps from severity: critical → high, warning → medium, info → low
  • If the signals array is empty → return empty recommendations (Gemini must not invent signals)
  • Output must be pure JSON — no markdown fences, no commentary

Gemini Model Chain

Tried in order. Next model is used on any error (rate limit, 503, deprecation):

#ModelRole
1gemini-2.5-flashPrimary — fastest, best quality
2gemini-2.0-flashSecondary fallback
3gemini-2.0-flash-liteFinal fallback

Config: temperature=0.4, max_output_tokens=4096, response_mime_type="application/json" (forces structured JSON, eliminates markdown stripping). If all three models fail, the function returns [] — no crash, no exception propagation.

Stage 5 — Enrichment & Persistence

File: recommendations/service.py → _enrich(rec, idx)

Each recommendation returned by Gemini is enriched with:

  • A stable UUID id (or passed through if Gemini provided one)
  • Priority normalized to low | medium | high (any unrecognized value → "medium")

The final blob is written to global_context["recommendations"] on the datasets table using a merge — other keys in global_context (step7, step8, segmentation, etc.) are preserved.

{
    "generated_at":   "2026-05-05T10:30:00+00:00",
    "snapshot_hash":  "a3f9b2c1d4e5f6g7",        // 16-char SHA1 prefix
    "signals":        [...],                        // all fired signals
    "recommendations": [...]                        // enriched list
}

No Forecast / Segmentation Case

The engine always works — even for brand-new datasets.

As long as the pipeline completed Step 8 (AI narrative report), detect_step8_insights fires and provides the report's section summaries as Gemini's primary evidence. Gemini derives domain-specific recommendations from those summaries.

Additionally, detect_no_forecast and detect_no_segmentation fire as info signals, prompting Gemini to recommend running those analyses and explain their specific business value for that category (Sales, HR, Operations, or Marketing).

Forecasting System

A multi-model competition engine that trains 7 forecasting models via rolling cross-validation, blends the two best when they agree, and produces a point forecast with confidence intervals stored in forecast_logs.

Full Pipeline: run_forecast_service()

File: backend/api/forecasting/service.py

Step 1 — Readiness Checks

Validated before any computation. Returns forecast_possible: False if any check fails:

  • Time column exists and ≥ 80% of values are parseable as dates
  • Target column exists and ≥ 60% of values are numeric
  • At least 3 valid (time, target) row pairs exist after parsing
  • All user-specified feature columns are present in the DataFrame

Step 2 — Series Preparation

Frequency Detection

Computed from the median gap between consecutive timestamps:

Median gapAssigned frequencySeason length
≈ 1 dayD (daily)7 (weekly cycle)
≈ 7 daysW (weekly)52
≈ 30 daysMS (month-start)12
≈ 90 daysQS (quarter-start)4

Processing Steps

  1. Parse time column to datetime; drop unparseable rows
  2. Convert target to float; handle nulls per missing_periods_policy (default: drop)
  3. Resample to detected frequency — sums values within each period, fills gaps
  4. IQR outlier capping — values beyond [Q1 − 1.5×IQR, Q3 + 1.5×IQR] are capped; capped points recorded as anomalies
  5. Log transform detection — applied if: all values > 0, coefficient of variation > 0.5, and log-space residuals have lower variance than raw-space. Predictions are exponentiated back before output.

Step 3 — Model Eligibility

Each model has a minimum required history. A model is eligible only if total_rows ≥ min_history + horizon. Ineligible models are skipped and logged.

Step 4 — Rolling Cross-Validation

Runs a 3-fold rolling backtest for each eligible model. Each fold uses the first N × (fold/folds) rows to train and predicts the next horizon steps.

Training is capped for expensive models to avoid O(n²) fitting time:

ModelCV train capReason
LightGBM / CatBoost1,095 rows3 years covers annual lag cycle
Prophet1,095 rowsNeeds 3 years for annual Fourier seasonality
SARIMAX500 rowsauto_arima AIC grid is slow

Metrics computed per fold (then averaged)

MetricDefinition
MAEMean Absolute Error — average absolute deviation
RMSERoot Mean Squared Error — penalizes large errors more
WAPEWeighted Absolute Percentage Error — MAE / mean(actuals)
MASEMean Absolute Scaled Error — relative to naive baseline

Model selection uses average rank across all 4 metrics — not just a single metric. This prevents a model that wins on MAE but loses badly on RMSE from being selected unfairly.

Step 5 — 70/30 Holdout Test

In parallel with CV, _evaluate_holdout_test() trains on the first 70% and evaluates on the last 30%. Used only for overfit diagnosis, not for model selection.

Fit Diagnosis (test_mae / cv_mae ratio)

RatioDiagnosisMeaning
< 0.70check_leakageTest is suspiciously easy — possible data leakage
0.70 – 1.20healthyModel generalizes well
1.20 – 1.50mild_overfitSlight generalization gap — monitor
1.50 – 2.00overfitSignificant gap — consider simpler model
> 2.00severe_overfitModel memorized training data — triggers recommendation signal

Step 6 — Model Selection

_select_best_model() ranks all successful models by their average rank across MAE, RMSE, WAPE, and MASE. Lower average rank = better overall performer.

The primary selection metric is MAE — used for tie-breaking and ensemble eligibility checks.

Step 7 — Ensemble Blending

If the runner-up model's MAE is within 25% of the winner's MAE, the two predictions are blended 50/50:

future_pred = (best_pred + partner_pred) / 2.0
ensemble_label = "model1+model2"   # e.g. "prophet+lightgbm"

When ensemble is active, native prediction intervals from the best model are discarded. Empirical intervals (from CV residuals) are used instead — because the residual distribution of the blend differs from either component model alone.

Step 8 — Forecast Generation

  1. Refit the best model (or both ensemble components) on the full dataset
  2. Build a future date frame of horizon periods at the detected frequency
  3. Call predict_with_intervals():
    • SARIMAX — native Bayesian prediction intervals from statsmodels
    • All other models — empirical intervals from CV residuals at 80% coverage
  4. If log-transform was applied, exponentiate predictions and intervals back
  5. Clamp lower bound to ≥ 0 if original series was non-negative
  6. Ensure every interval brackets its point forecast — adjust if not, and count adjustments

All 7 Forecasting Models

Naive
NaiveForecaster
Repeats the last observed value for all future periods. Serves as the absolute baseline. Min: 3 rows. Always eligible.
Seasonal Naive
SeasonalNaiveForecaster
Repeats the last full seasonal cycle. Surprisingly strong on data with regular weekly / monthly patterns. Min: 2 full cycles.
ETS (Holt-Winters)
ETSForecaster
Exponential smoothing with additive trend + additive seasonality. Uses statsmodels ExponentialSmoothing with auto-optimization. Falls back to non-seasonal if series is short. Min: 10 rows.
SARIMAX
SarimaxForecaster
ARIMA with seasonal terms + optional exogenous features. Orders selected via AIC grid search over p∈{0,1,2}, d∈{0,1}, q∈{0,1,2} with a 20-second timeout. Falls back to heuristic on timeout. Provides native Bayesian intervals. Min: 20 rows.
CatBoost
CatBoostForecaster
Gradient boosting tree with recursive multi-step forecasting. Uses 30 lag/rolling/calendar features. Handles non-linearity and interactions naturally. Min: 30 rows. CV cap: 1,095 rows.
LightGBM
LightGBMForecaster
Identical feature set to CatBoost but uses a different histogram-based boosting algorithm. Generally faster to train. Often wins on large datasets. Min: 30 rows. CV cap: 1,095 rows.
Prophet
ProphetForecaster
Facebook's additive decomposition model. Handles multiple simultaneous seasonalities (daily, weekly, yearly) automatically via Fourier series. Exogenous regressors added for user-selected feature columns. Min: 20 rows. CV cap: 1,095 rows.
